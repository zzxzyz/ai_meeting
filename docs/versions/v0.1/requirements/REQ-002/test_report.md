## 关键发现

### 已验证的功能点

1. **会议号唯一性机制**: `meeting_number` 字段有 UNIQUE 索引约束，代码层面最多重试 3 次，数据库层面兜底。

2. **会议状态机**:
   - 创建 → `IN_PROGRESS` (映射自 `MeetingStatus.ACTIVE`)
   - `IN_PROGRESS` → `ENDED`（仅创建者操作）
   - `ENDED` 为终态

3. **参与者幂等性**: `findActiveParticipant` 通过检查 `left_at IS NULL` 实现幂等，避免重复添加。

4. **durationSeconds 计算**:
   - 进行中：实时计算（当前时间 - startedAt）
   - 已结束：固定值（endedAt - startedAt）
   - 精度：秒级（Math.floor），符合要求

5. **WebSocket 通知**: 通过 `meeting:{meetingId}` 房间机制，确保只有加入房间的参与者收到通知。

### 需要关注的事项

1. **会议号格式说明**: 数据库存储为连续 9 位数字，API 契约返回 9 位纯数字字符串，前端负责格式化显示（`XXX-XXX-XXX`）。两者均在代码中实现。

2. **analysis.md 与 api_contract.md 差异**:
   - `analysis.md` 中状态使用 `active`/`ended`
   - `api_contract.md` 中状态使用 `IN_PROGRESS`/`ENDED`
   - 后端 `mapStatus()` 函数正确转换 DB 内部值 → API 响应值
   - **建议**: 后续文档应统一状态枚举命名

3. **getMeetingDetail 访问权限**: 分析文档中描述"任何已登录用户均可查询"，但 API 契约和代码实现均为"只有参与者或创建者可查询"（返回 403）。已按代码实现进行测试。

4. **会议结束 WebSocket 依赖**: 客户端需主动调用 `join_meeting_room` WebSocket 事件加入会议房间才能收到推送，这是正常设计但客户端需确保在会议页面加载时建立 WebSocket 连接。

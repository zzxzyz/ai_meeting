# Agent Team 协作机制优化 - 实施路线图

## 文档概述

本文档汇总了 Agent Team 协作机制优化方案的完整实施计划，基于 REQ-001 开发过程中暴露的问题，提出分阶段的优化路径。

**文档目的**:
- 为用户提供清晰的优化方案概览
- 明确各阶段的目标、措施和时间表
- 指导后续需求开发中的协作改进

**文档版本**: 1.0
**创建日期**: 2026-02-15
**创建者**: team-lead
**适用范围**: 视频会议系统项目的 Agent Team 协作

## 一、背景与问题总结

### 1.1 REQ-001 开发回顾

**成功之处** ✅:
- 高质量交付：6份文档(~90KB)、24个代码文件、52个测试用例
- 测试覆盖率 87.5%，代码复用率 100%(Electron端)
- 并行协作有效：文档、实现、测试阶段均实现并行开发
- 开发周期短：约2小时完成全部任务

**暴露的问题** ⚠️:
1. **任务分配机制缺陷**: 依赖agents自主领取，使用了 `-2` 后缀的动态agents
2. **并发控制缺失**: 无任务锁机制，可能导致重复领取
3. **角色权限控制缺失**: 缺乏RBAC强制机制
4. **任务状态不同步**: agents未及时更新任务状态

**详细分析**: 见 `docs/team_collaboration/REQ-001_retrospective.md`

### 1.2 核心问题根因

| 问题类别 | 根本原因 | 影响范围 | 优先级 |
|---------|---------|---------|--------|
| 任务分配机制 | TaskList无过滤能力，agents看到所有任务 | 角色匹配度、分配准确性 | **P0** |
| 并发控制 | 无version字段，无状态转换约束 | 任务重复率、状态一致性 | **P0** |
| 角色权限 | 无RBAC系统，依赖agents自觉性 | 职责清晰度、专业性 | **P1** |

### 3.2 核心措施

#### 措施1: 优化任务描述格式

**规范文档**: `docs/team_collaboration/task_creation_guidelines.md`

**关键要求**:
- Subject 格式: `[角色名] 任务简述`
- Description 开头: `⚠️ **本任务仅限 XXX 领取**`
- 包含完整的任务目标、具体工作、交付物、参考文档

**示例**:
```
Subject: [backend-leader] REQ-002 后端实现 - 会议管理服务
Description:
⚠️ **本任务仅限 backend-leader 领取**

**任务目标**: 实现完整的会议管理后端服务...
```

#### 措施2: Team-lead 主动分配

**工作流程**: `docs/team_collaboration/team_lead_workflow.md`

**标准流程**:
1. 创建任务后立即使用 TaskUpdate 设置 owner
2. 不依赖agents自主领取
3. 发送 SendMessage 通知对应agent

**示例**:
```javascript
// 1. 创建任务
TaskCreate({ subject: "[backend-leader] REQ-002 后端实现", ... })  // Task #5

// 2. 立即分配
TaskUpdate(5, { owner: "backend-leader" })

// 3. 发送通知
SendMessage({
  recipient: "backend-leader",
  content: "已为你分配 Task #5...",
  summary: "新任务分配: Task #5"
})
```

#### 措施3: 监控和纠正

**监控机制**:
- Team-lead 每15-30分钟检查TaskList
- 发现跨角色领取时立即纠正
- 及时更新任务状态

## 四、阶段2: 工具层优化 (中期实施)

### 4.1 目标与指标

**优化目标**:
- 从系统层面防止角色混淆
- 实现并发控制机制
- 降低team-lead手动管理负担

**实施条件**:
- ✅ 阶段1核心指标全部达标
- ✅ Team-lead单任务操作耗时 >5分钟，或主动提出需要自动化

### 4.2 核心功能

#### 功能1: 结构化任务定义

**新增字段**:
- `requiredRole`: 指定任务的角色要求 (可选字符串)
- `version`: 任务版本号 (整数，初始为1)
- `taskType`: 任务类型枚举 (可选)

**Schema 设计**: `task_schema_v2.json` (8.4KB) ✅

**向后兼容**: 所有新字段都是可选或有默认值

#### 功能2: 乐观锁机制

**实现方式**:
- TaskUpdate 支持 `expectedVersion` 参数
- 如果版本不匹配，拒绝更新并返回错误
- 成功更新后，version自动递增

**示例**:
```javascript
// Agent A 读取任务
const task = TaskGet(5);  // { id: 5, version: 1 }

// Agent A 尝试更新
TaskUpdate(5, { status: "in_progress", expectedVersion: 1 })  // 成功，version变为2

// Agent B 同时尝试更新（使用旧版本）
TaskUpdate(5, { status: "in_progress", expectedVersion: 1 })  // 失败：版本冲突
```

#### 功能3: 角色匹配验证

**实现方式**:
- TaskUpdate 设置owner时，检查调用者角色是否匹配任务的`requiredRole`
- 如果不匹配，返回警告或拒绝操作
- team-lead可使用 `forceAssign` 参数强制分配

**示例**:
```javascript
// architect尝试领取后端任务
TaskUpdate(5, { owner: "architect" })
// 错误：Task #5 requires role 'backend-leader', but caller is 'architect'

// team-lead强制分配
TaskUpdate(5, { owner: "architect", forceAssign: true })  // 成功
```

#### 功能4: 角色过滤

**实现方式**:
- TaskList 支持 `roleFilter` 参数
- agents 只看到自己角色的任务 + 无角色限制的任务

**示例**:
```javascript
TaskList({ roleFilter: "backend-leader" })
// 返回：requiredRole为"backend-leader"或无限制的任务
```

#### 功能5: 状态转换验证

**允许的转换**:
- `pending` → `in_progress` (领取任务)
- `in_progress` → `completed` (完成任务)
- `in_progress` → `pending` (放弃任务)

**禁止的转换**:
- `pending` → `completed` (跳过进行中状态)
- `completed` → `in_progress` (已完成任务不可重新打开)

## 五、阶段3: RBAC和智能推荐 (长期规划)

### 5.1 目标与指标

**优化目标**:
- 建立完善的RBAC权限系统
- 实现智能任务推荐机制
- 支持任务优先级管理

**实施条件**:
- ✅ 阶段2实施完成
- ✅ 至少完成3-4个需求，积累足够协作数据
- ✅ 团队协作模式相对稳定

### 5.2 核心功能

#### 功能1: RBAC配置系统

**角色权限映射**:
```json
{
  "product-manager": ["documentation", "design"],
  "architect": ["documentation", "design", "architecture"],
  "backend-leader": ["backend"],
  "frontend-leader": ["frontend"],
  "client-leader": ["client"],
  "test-leader": ["testing"],
  "devops-leader": ["devops"],
  "team-lead": ["*"]  // 全权限
}
```

**任务类型枚举**:
- documentation, design, architecture, backend, frontend, client, testing, devops

**配置文件**: `rbac_template.json` ✅

#### 功能2: 任务领取审批流程

**跨角色领取审批**:
1. agent尝试领取不匹配角色的任务
2. 触发审批请求，发送给team-lead
3. team-lead批准或拒绝
4. 批准后完成任务领取

**消息格式**:
- `task_assignment_request`: 审批请求
- `task_assignment_response`: 审批响应

#### 功能3: 智能任务推荐

**推荐算法**:
- 角色匹配度: 40%
- 历史完成记录: 30%
- 任务优先级: 20%
- 负载平衡: 10%

**推荐时机**:
- agent空闲时(idle_notification)
- team-lead主动推荐最适合的任务

#### 功能4: 任务优先级系统

**优先级枚举**: high / medium / low

**影响规则**:
- 推荐算法优先推荐高优先级任务
- TaskList可按优先级排序
- team-lead可设置任务优先级

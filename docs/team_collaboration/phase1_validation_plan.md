## 一、验证目标

### 1.1 核心验证问题

**主要问题**: 阶段1的优化措施(主动分配机制 + 任务描述规范)能否有效解决 REQ-001 中出现的协作问题?

**具体验证点**:
1. ✅ 主动分配机制能否消除跨角色领取问题?
2. ✅ 任务描述规范能否减少沟通成本?
3. ✅ 角色匹配度能否达到 >90% 的目标?
4. ✅ 任务重复率能否保持 0%?
5. ✅ Team-lead 的管理负担是否可接受?

### 1.2 验证假设

| 假设ID | 假设内容 | 验证指标 | 目标值 |
|-------|---------|---------|-------|
| H1 | 主动分配可以完全避免跨角色领取 | 角色匹配率 | 100% |
| H2 | 任务描述规范可以减少 30% 的澄清询问 | 问题询问消息占比 | ≤10% |
| H3 | 主动分配不会显著增加 team-lead 工作量 | Team-lead 操作耗时 | ≤5分钟/任务 |
| H4 | 任务重复领取可以完全避免 | 任务重复率 | 0% |
| H5 | 整体协作效率不会因流程增加而下降 | 平均任务完成时间 | ≤30分钟 |

## 二、验证指标体系

### 2.1 核心指标 (P0)

这些指标直接反映阶段1优化方案的效果,必须达标。

| 指标名称 | 计算公式 | 目标值 | 数据来源 | 监控频率 |
|---------|---------|-------|---------|---------|
| **角色匹配率** | 角色匹配任务数 / 总任务数 × 100% | **>90%** | REQ-002 任务记录表 | 实时 |
| **任务重复率** | 重复领取任务数 / 总任务数 × 100% | **0%** | REQ-002 任务记录表 | 实时 |
| **任务分配准确性** | (总任务数 - 重新分配次数) / 总任务数 × 100% | **100%** | REQ-002 任务记录表 | 实时 |

**达标标准**:
- ✅ 全部达标: 阶段1优化方案验证成功,可进入阶段2
- ⚠️ 部分达标: 需要调整优化措施,继续在 REQ-003 中验证
- ❌ 全部不达标: 阶段1方案失败,需要重新设计

### 2.2 效率指标 (P1)

这些指标反映优化方案对整体效率的影响。

| 指标名称 | 计算公式 | 目标值 | 数据来源 | 监控频率 |
|---------|---------|-------|---------|---------|
| **平均任务完成时间** | Σ(任务用时) / 任务数 | ≤30分钟 | REQ-002 任务记录表 | 需求结束后 |
| **问题询问消息占比** | 问题询问消息数 / 总消息数 × 100% | ≤10% | REQ-002 沟通记录 | 需求结束后 |
| **平均响应时间** | Σ(响应时间) / 消息数 | ≤15分钟 | REQ-002 沟通记录 | 需求结束后 |

**警示阈值**:
- 🔴 平均任务完成时间 >45 分钟: 优化方案可能增加了负担
- 🔴 问题询问占比 >20%: 任务描述规范未起作用

### 2.3 管理负担指标 (P1)

评估 team-lead 的工作负担是否可接受。

| 指标名称 | 计算公式 | 目标值 | 数据来源 | 监控频率 |
|---------|---------|-------|---------|---------|
| **Team-lead 单任务操作耗时** | 创建+分配+通知的总耗时 / 任务数 | ≤5分钟 | Team-lead 自我记录 | 需求结束后 |
| **Team-lead 干预次数** | 纠正跨角色 + 重新分配 + 手动更新状态 | ≤2次 | REQ-002 问题记录表 | 实时 |

**可接受标准**:
- ✅ 单任务操作耗时 ≤5 分钟: 管理成本可接受
- ⚠️ 5-10 分钟: 勉强可接受,需要优化工具
- ❌ >10 分钟: 管理成本过高,需要自动化

### 2.4 体验指标 (P2)

收集各 agent 的主观体验反馈。

| 体验维度 | 评估方法 | 收集方式 |
|---------|---------|---------|
| 任务描述清晰度 | 5分制评分 | 需求结束后调查 |
| 任务分配合理性 | 5分制评分 | 需求结束后调查 |
| 沟通效率满意度 | 5分制评分 | 需求结束后调查 |
| 协作流程顺畅度 | 5分制评分 | 需求结束后调查 |

**评分标准**:
- 5分: 非常满意
- 4分: 满意
- 3分: 一般
- 2分: 不满意
- 1分: 非常不满意

**目标**: 所有维度平均分 ≥4 分

## 四、问题识别机制

### 4.1 自动检测规则

**规则1: 跨角色领取检测**

**触发条件**:
- 任务的 subject 中包含 `[角色A]`
- 但 owner 被设置为 `角色B`
- 且 `角色A ≠ 角色B`

**检测时机**: TaskUpdate 设置 owner 时

**检测方法**:
```javascript
// Team-lead 在分配任务后立即检查
TaskList()
// 检查每个任务的 subject 和 owner 是否匹配

// 示例:
// Task #5 subject: "[backend-leader] REQ-002 后端实现"
// Task #5 owner: "architect"  ❌ 不匹配! 触发告警
```

**告警动作**:
1. 记录到问题表: 问题类型 = 跨角色领取
2. Team-lead 立即纠正:
   - 发送消息给错误的 owner,要求停止
   - 重新分配给正确的角色
   - 通知正确的角色开始

**责任人**: Team-lead (主动检查)

---

**规则2: 任务重复领取检测**

**触发条件**:
- 任务的 owner 被修改超过 1 次
- 或任务被标记为 in_progress 超过 1 次

**检测时机**: TaskUpdate 修改 owner 或 status 时

**检测方法**:
```javascript
// Team-lead 在每次 TaskList 时检查
// 记录每个任务的 owner 修改次数

// 如果发现某个任务的 owner 被修改 >1 次,触发告警
```

**告警动作**:
1. 记录到问题表: 问题类型 = 任务重复领取
2. Team-lead 立即协调:
   - 与涉及的 agents 沟通
   - 确定最终负责人
   - 明确只有一个 agent 继续工作

**责任人**: Team-lead (主动检查)

---

**规则3: 任务状态不同步检测**

**触发条件**:
- 任务处于 in_progress 状态超过 60 分钟
- 或 agent 报告任务完成,但状态仍为 in_progress

**检测时机**:
- 定期检查 (每 30 分钟)
- 收到 agent 完成通知时

**检测方法**:
```javascript
// Team-lead 定期 TaskList 检查
// 对比任务状态和实际进度

// 如果发现长时间 in_progress 但无进展,触发询问
```

**告警动作**:
1. 记录到问题表: 问题类型 = 状态不同步
2. Team-lead 询问 agent 进度:
   - 如已完成但未更新,提醒更新状态
   - 如遇到阻塞,协助解决

**责任人**: Team-lead (定期检查)

## 六、阶段2启动决策标准

### 6.1 决策矩阵

阶段2优化方案(自动化工具)需要较大投入,因此需要明确的启动标准。

| 决策场景 | 核心指标表现 | 管理负担 | 决策 |
|---------|------------|---------|-----|
| **场景A: 完美达标** | 角色匹配率 100%, 任务重复率 0%, 分配准确性 100% | Team-lead 单任务耗时 ≤5分钟 | ❌ **不启动阶段2**<br>当前方案已足够好,无需自动化 |
| **场景B: 达标但负担重** | 角色匹配率 ≥90%, 任务重复率 0%, 分配准确性 100% | Team-lead 单任务耗时 >5分钟 | ✅ **启动阶段2**<br>效果好但效率低,需自动化提效 |
| **场景C: 部分达标** | 角色匹配率 70%-89%, 或任务重复率 1%-5% | 任意 | ⚠️ **暂缓阶段2**<br>先优化阶段1方案,在 REQ-003 中继续验证 |
| **场景D: 全不达标** | 角色匹配率 <70%, 或任务重复率 >5% | 任意 | ❌ **不启动阶段2**<br>阶段1方案失败,需重新设计 |

### 6.2 启动标准详解

**必要条件** (必须同时满足):
1. ✅ 角色匹配率 ≥90%
2. ✅ 任务重复率 = 0%
3. ✅ 任务分配准确性 = 100%

**充分条件** (满足任一即可启动阶段2):
1. Team-lead 单任务操作耗时 >5 分钟
2. Team-lead 管理负担评估为"显著增加"
3. Team-lead 主动提出需要自动化工具
4. 需求规模预期显著增加 (如 REQ-003 预计有 >15 个任务)

**决策时机**: REQ-002 数据分析完成后

**决策者**: Team-lead (基于数据和体验)

## 九、风险与应对

### 9.1 数据收集风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| Team-lead 忘记记录数据 | 数据不完整,无法准确计算指标 | - 设置定期提醒 (每完成3-5个任务更新一次)<br>- Test-leader 中期检查提醒 |
| 数据记录不一致 | 分析结果不准确 | - 使用标准模板<br>- 明确字段定义<br>- Test-leader 检查数据质量 |
| 体验调查回复率低 | 无法全面评估体验 | - 明确说明调查目的<br>- 简化问卷内容<br>- 适当催促 |

### 9.2 验证执行风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| REQ-002 规模与 REQ-001 差异过大 | 对比分析不准确 | - 标准化指标计算方法<br>- 分析时考虑规模因素 |
| REQ-002 出现 REQ-001 未遇到的问题 | 验证不充分 | - 灵活调整验证计划<br>- 记录新问题并分析 |
| Team-lead 未严格执行优化方案 | 验证无效 | - 明确执行要求<br>- Test-leader 初期检查执行情况 |

### 9.3 决策风险

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 指标达标但实际体验差 | 启动阶段2后效果不佳 | - 综合考虑量化指标和主观体验<br>- 体验评分设置否决权 (平均分 <3 不启动) |
| 过早启动阶段2 | 投入自动化但收益不明显 | - 严格执行必要条件检查<br>- 充分条件需 Team-lead 确认 |
| 错过启动时机 | Team-lead 负担过重,影响协作效率 | - 及时做出决策 (REQ-002 完成后 1 天内)<br>- 允许 Team-lead 主动提出需求 |
